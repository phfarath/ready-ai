# Security Policy

## Supported Scope

Security reports are welcome for the code in this repository, especially issues involving:

- credential handling
- cookie/session handling
- command execution risks
- prompt injection or unsafe browser automation behaviors
- insecure file output or data exposure

This project is early stage. Support and response times are best effort.

## How to Report a Vulnerability

Please do not report security vulnerabilities in public GitHub issues.

Instead:

1. Contact the maintainer privately through the repository owner's GitHub profile or configured repository contact channel.
2. Include a clear description of the issue, impact, affected files or flows, and reproduction steps.
3. If possible, include a minimal proof of concept without exposing real credentials or user data.

If a dedicated private reporting channel is added later, this document should be updated to point to it directly.

## What to Include

Please include:

- vulnerability type
- impact
- affected components
- reproduction steps
- suggested mitigation, if known

## Disclosure Policy

- We will try to confirm the issue and assess impact.
- We may ask for additional reproduction details.
- Please allow reasonable time for investigation and remediation before public disclosure.

## Safe Handling Expectations

When testing or reporting issues:

- never include real API keys, cookies, or passwords in public reports
- avoid testing against systems you do not own or have permission to assess
- redact screenshots and logs that contain sensitive data
